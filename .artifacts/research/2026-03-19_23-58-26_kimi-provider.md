## Validated KIMI Facts
- KIMI should be integrated here as an Anthropic-like provider, not as an OpenAI chat-completions provider.
- Base URL: `https://api.kimi.com/coding`
- Endpoint shape: `/v1/messages`
- Auth: Bearer token sourced from `KIMI_API_KEY`
- Model ID to use in examples and initial wiring: `kimi-coding`
- Reasoning should be enabled.
- Published limits for the intended integration pass: 128K context window and 16K output tokens.
- These facts imply a shared `anthropic_like` runtime is the right implementation direction, but not necessarily a drop-in reuse of the current `async_anthropic::Client` transport layer.

## Patterns Found
- Provider selection fan-out:
  - `src/types/handlers.rs:L5-L10` defines the provider variant list.
  - `src/looper.rs:L66-L129` and `src/looper_stream.rs:L84-L150` each match on every provider variant and construct the concrete handler for that mode.
- Provider module pairing:
  - `src/services/handlers/mod.rs:L1-L8` exports one streaming/non-streaming pair for Anthropic, OpenAI Completions, OpenAI Responses, and Gemini.
- Anthropic request construction pattern:
  - `src/services/handlers/anthropic.rs:L60-L67` and `src/services/handlers/anthropic_non_streaming.rs:L53-L60` both build requests with `CreateMessagesRequestBuilder`, `.model(...)`, `.system(...)`, `.messages(...)`, `.tools(...)`, `.max_tokens(16384)`, and `.thinking(Thinking::Adaptive)`.
- Anthropic tool-result encoding pattern:
  - `src/services/handlers/anthropic.rs:L209-L220` appends `MessageRole::User` with `MessageContent::ToolResult(ToolResultBuilder::default()...)` after each tool completes.
  - `src/services/handlers/anthropic_non_streaming.rs:L127-L138` appends the same tool-result message shape in non-streaming mode.
- Anthropic history storage pattern:
  - `src/services/handlers/anthropic.rs:L243-L261` restores and returns `MessageHistory::Messages`.
  - `src/services/handlers/anthropic_non_streaming.rs:L175-L193` restores and returns `MessageHistory::Messages`.
- Anthropic tool schema mapping pattern:
  - `src/services/handlers/anthropic.rs:L264-L268` and `src/services/handlers/anthropic_non_streaming.rs:L202-L206` both convert `Vec<LooperToolDefinition>` with `.map(|t| t.into())`.
  - `src/mapping/tools/anthropic.rs:L4-L11` defines that `into()` conversion.
- Explicit env handling pattern differs by provider:
  - Gemini constructors read env vars in-repo at `src/services/handlers/gemini.rs:L38-L47` and `src/services/handlers/gemini_non_streaming.rs:L32-L41`.
  - Anthropic constructors do not read env vars in-repo; they call `Client::default()` at `src/services/handlers/anthropic.rs:L40` and `src/services/handlers/anthropic_non_streaming.rs:L36`.
  - The external Anthropic SDK default client resolves `ANTHROPIC_API_KEY` and sets its base URL in `/Users/tuna/.cargo/git/checkouts/async-anthropic-89ac3a957c31bf38/bddf804/src/client.rs:L59-L89`.
- KIMI compatibility is body-shape compatible with Anthropic, but current transport wiring is not fully reusable as-is:
  - `src/services/handlers/anthropic.rs:L40` and `src/services/handlers/anthropic_non_streaming.rs:L36` currently hardcode `Client::default()`.
  - `/Users/tuna/.cargo/git/checkouts/async-anthropic-89ac3a957c31bf38/bddf804/src/messages.rs:L19-L40` already matches the desired `/v1/messages` endpoint shape if the base URL is set to `https://api.kimi.com/coding`.
  - `/Users/tuna/.cargo/git/checkouts/async-anthropic-89ac3a957c31bf38/bddf804/src/client.rs:L121-L128` currently sends `x-api-key`, while KIMI requires Bearer auth from `KIMI_API_KEY`.
  - Result: the request/response loop, history model, and tool mapping are reusable, but the HTTP client construction likely needs an `anthropic_like` abstraction or a KIMI-specific transport adapter.
- KIMI feature shape for this crate should mirror Anthropic, not Gemini or OpenAI:
  - `MessageHistory::Messages` remains the correct history representation.
  - `src/mapping/tools/anthropic.rs:L4-L11` is the right tool-schema mapping to reuse.
  - Streaming and non-streaming behavior should stay paired, just as Anthropic and Gemini already do.
- Example env loading pattern:
  - `examples/cli.rs:L16` and `examples/cli_non_streaming.rs:L16` call `dotenv::dotenv().ok()` before constructing loopers.
- Current repository scan pattern for KIMI:
  - `rg -n "KIMI|kimi" . --hidden` returned no source references.

## Implementation Implications
- Add a new `Handlers::Kimi(Model<'a>)` variant in `src/types/handlers.rs` and thread it through both builder match statements in `src/looper.rs` and `src/looper_stream.rs`.
- Prefer factoring the Anthropic request loop into a shared internal `anthropic_like` runtime rather than copying the full handler logic into separate `kimi` files.
- Keep the outbound tool mapping on the Anthropic path unless KIMI later proves to diverge from Anthropic `Tool` schema.
- Do not point a KIMI base URL at `https://api.kimi.com/coding/v1` when using the current Anthropic message path helper, because `/v1/messages` is already appended by the client layer.
- `.env.example`, `README.md`, and the architecture docs will need updates once wiring lands, because the repository currently documents only `OPENAI_API_KEY` in `.env.example` and does not mention `KIMI` in public provider lists.
